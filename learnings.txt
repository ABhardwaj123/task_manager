in this doc , I will write what I have done in each file
and some learnings and decisions I learnt / made at each step

production workflow:



--.env-- 
store some private/secret info that is not supposed to made public.
not to be pushed to github either
my jwt secret key is used to sign / create JWT token
JWT token is given to each user so that next time they log in , they can be identified by JWT



--database.py--
SQLAlchemy sits between database and Python. Helps python to talk to database in terms of classes 
created engine that prepares SQLAlchemy for communication
creating a sessionmaker that makes sessions (sessions to talk to database)
Database sessions are created only when we need to talk to the database
declaring a Base (Base tells you which all classes are actual tables/models)

get_db() is a dependency that FastAPI routes will use to get a session






--main.py--
created a fastAPI instance 
Base.metadata.create_all(bind=engine) -> this line actually creates tables in postgreSQL
linked router object from /routes/auth.py (router object groups the auth endpoints together)






--models.py--
basically how data is stored in postgreSQL
has the database models for user and tasks for pydantic classes
importing Base from database that helps to recognize both the classes of User and Task as schemas







--auth.py--
we hash the password entered by the user and store only the hashed password in the database
we hash it using bcrypt
another verify_password function , so that next time user logins , we are able to confirm if 
correct password is entered or not

user gets JWT token after logging in and then on future requests , token provides identity






--schemas.py--
these schemas tells the shape of data for a specific API
how data enters and leaves the API






--dependencies.py--
say a user goes to some route , now we have to check inside every route if the user is logged in or not
to remove that redundancy , we write the code of checking if user is logged in or not once in this file

use OAuth2PasswordBearer -> this extracts the JWT token from the authorization header
Depends in function parameters -> it first runs the dependencies which are as parameters of the function
        after that the function runs

if token is valid , full user object is returned to the route






--/routes/auth.py--
added two endpoints of user registration and user login
taking care if at every point valid email is there , valid password is there etc
learnt some syntax related to fetching data from database

created a router instance -> it groups related endpoints so that they can be plugged into main app as one unit





--/routes/tasks.py
added endpoints for tasks 
1. get all the tasks
2. add a task
3. update a task
4. delete a task
